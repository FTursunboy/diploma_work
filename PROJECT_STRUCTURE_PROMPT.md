# Промт для ИИ: понятная структура проекта для дипломного руководителя

Ниже готовый промт, составленный по реальной структуре этого проекта. Он специально настроен так, чтобы ИИ объяснял систему простым, академически понятным языком, без лишней технической перегрузки.

## Готовый промт

```text
Изучи описание проекта ниже и создай идеальную, очень понятную и аккуратную структуру проекта для показа дипломному руководителю.

Это не техническая документация для разработчика.
Это объяснение проекта для человека, который должен быстро понять:
- что это за система;
- для кого она сделана;
- какую задачу решает;
- из каких логических частей состоит;
- как пользователь работает с системой;
- как внутри проходит путь документа от загрузки до поиска;
- почему структура проекта логична и удобна.

Важно:
- пиши максимально простым, ясным и академически понятным языком;
- не перегружай ответ программистскими терминами;
- если используешь термин, сразу объясняй его простыми словами;
- не делай сухой список файлов без смысла;
- показывай не просто папки, а роль каждой части проекта;
- объясняй структуру так, чтобы её понял дипломный руководитель, а не разработчик;
- текст должен выглядеть как качественное, аккуратное описание дипломной системы;
- стиль должен быть официальный, понятный, красивый и уверенный;
- ответ должен быть одновременно простым и подробным;
- не упрощай до уровня “в проекте есть backend и frontend”, а раскрой смысл каждого блока;
- не уходи в код и низкоуровневые детали реализации;
- не описывай служебные папки как основную архитектуру проекта, если они не важны для понимания системы.

Что нужно выдать в ответе:

1. Название проекта и его краткое понятное назначение
2. Общая идея системы в 1 небольшом абзаце
3. Основные функции системы
4. Логическая структура проекта по крупным блокам
5. Физическая структура проекта по папкам и файлам, но с простым объяснением назначения каждой части
6. Путь работы документа внутри системы: загрузка -> сохранение -> обработка -> разбиение текста -> сохранение в базе -> поиск -> просмотр результата
7. Роли пользователей и что может делать каждая роль
8. Как устроена пользовательская часть проекта
9. Как устроена серверная часть проекта
10. Как устроено хранение данных
11. Почему такая структура проекта удобна и логична
12. Короткий вывод для дипломного руководителя

Формат ответа обязательно такой:

1. Сначала дай раздел "Краткое описание проекта".
2. Потом раздел "Основные возможности системы".
3. Потом раздел "Логическая структура проекта".
4. Потом раздел "Структура папок и файлов проекта".
5. Потом раздел "Как работает система".
6. Потом раздел "Роли пользователей".
7. Потом раздел "Почему структура проекта удачная".
8. Потом раздел "Итог".

Требования к подаче структуры:
- сначала показывай крупные смысловые блоки;
- потом внутри них перечисляй файлы и папки;
- возле каждого важного файла или папки обязательно пиши простое объяснение;
- если есть второстепенные, служебные или устаревшие элементы, помечай их как вспомогательные и не делай на них главный акцент;
- делай структуру визуально чистой и аккуратной;
- используй понятные подзаголовки;
- не делай ответ слишком коротким;
- не делай ответ слишком техническим;
- ответ должен быть готовым для вставки в диплом или для показа руководителю.

Вот описание проекта, по которому нужно строить структуру:

Проект представляет собой веб-систему для работы с электронными документами и книгами в форматах PDF и DOCX.
Система позволяет загружать документы, извлекать из них текст, автоматически разбирать текст на абзацы, предложения и слова, сохранять результаты в базе данных и выполнять интеллектуальный поиск по содержимому документов.

Система включает веб-интерфейс и серверную часть.
Пользователь через браузер может:
- зарегистрироваться;
- войти в систему;
- просматривать список документов;
- искать документы по библиографическим данным;
- открывать документ и смотреть его полный текст;
- смотреть структуру текста по абзацам, предложениям и словам;
- выполнять поиск по слову, предложению, абзацу или фразе;
- использовать дополнительные инструменты анализа текста, такие как wordlist, concordance, n-grams и поиск по n-граммам;
- скачивать исходный файл.

В системе есть роли пользователей:
- user: просмотр документов, поиск, скачивание;
- moderator: все возможности user + загрузка новых документов;
- admin: все возможности moderator + управление ролями пользователей и удаление документов.

Основная серверная логика построена на FastAPI.
Проект имеет маршруты для:
- авторизации и регистрации;
- работы с документами;
- поиска;
- аналитических инструментов;
- отображения HTML-страниц.

Структура серверной части логически разделена:
- main.py — точка запуска приложения;
- routers/ — маршруты и API-эндпоинты;
- services/ — основная бизнес-логика;
- services/search/ — отдельные сервисы поиска по словам, предложениям, абзацам и фразам;
- database.py — модели базы данных и инициализация подключения;
- auth.py — логика токенов, ролей и паролей;
- parser.py, splitter.py, search.py — вспомогательные модули для совместимости и обработки текста.

В проекте есть пользовательский веб-интерфейс:
- web/index.html — основная страница работы с документами и поиском;
- web/login.html — вход;
- web/register.html — регистрация;
- web/admin.html — административная панель;
- web/static/ — JavaScript, CSS, favicon и логика интерфейса.

Система хранения данных устроена так:
- documents — основная информация о документе и полный текст;
- paragraphs — абзацы документа;
- sentences — предложения документа;
- words — слова документа;
- users — пользователи, их роли, e-mail и данные авторизации.

После загрузки файла система:
- сохраняет файл в storage/uploads;
- извлекает текст из PDF или DOCX;
- очищает и нормализует текст;
- разбивает текст на абзацы, затем на предложения, затем на слова;
- сохраняет структурированные данные в базу;
- делает документ доступным для просмотра и поиска.

Для запуска проекта используются:
- Dockerfile;
- docker-compose.yml;
- pyproject.toml.

В проекте также могут встречаться служебные элементы, которые не нужно делать центральной частью описания:
- storage/ как техническое хранилище файлов и базы;
- __pycache__/ как автоматически создаваемые служебные файлы;
- .venv/ как локальное окружение разработки;
- app/ как остаточная или неосновная директория, не являющаяся центром актуальной архитектуры.

Очень важно:
Сделай ответ не как набор технических заметок, а как образцовое объяснение архитектуры дипломного проекта.
Представь, что это описание будет читать научный руководитель, который хочет увидеть порядок, логику, структуру и зрелость проекта.
Пиши так, чтобы после прочтения у него сложилось впечатление, что проект хорошо организован, продуман и готов к защите.
```

## Зачем этот промт работает

- Он заставляет ИИ объяснять проект человеческим языком, а не сухим языком разработчика.
- Он задаёт чёткую структуру ответа, поэтому результат не расползается.
- Он опирается на реальный состав этого репозитория: маршруты, сервисы, веб-часть, роли, поиск, инструменты анализа и хранение данных.
- Он специально уводит ИИ от лишнего внимания к служебным и второстепенным папкам.
